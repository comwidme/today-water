# **오늘물줘 MVP TDD 테스트 설계**

## **1\. 목적**

이 문서는 오늘물줘 MVP를 구현하기 전, 핵심 기능을 테스트 기준으로 먼저 정의하기 위한 문서다.  
Cursor는 구현 시 이 문서를 참고해 테스트 가능한 단위로 기능을 작성한다.

MVP 테스트 목표는 다음과 같다.

1. 식물 등록이 올바르게 동작하는지 검증한다.  
2. 다음 물주기 날짜 계산이 정확한지 검증한다.  
3. 오늘/지연/예정 상태 계산이 정확한지 검증한다.  
4. 물주기 완료 시 데이터가 올바르게 갱신되는지 검증한다.  
5. LocalStorage 저장·불러오기가 안정적으로 동작하는지 검증한다.

## **5\. TDD 진행 원칙**

1. 실패하는 테스트를 먼저 작성한다.  
2. 테스트를 통과하는 최소 구현만 작성한다.  
3. 중복 로직을 유틸 함수로 분리한다.  
4. UI 테스트보다 날짜·상태·저장 로직 테스트를 우선한다.  
5. PRD에 없는 기능은 테스트도 작성하지 않는다.

# **6\. 단위 테스트 설계**

## **6.1 날짜 유틸 테스트**

### **대상 함수**

getTodayDateString()  
calculateNextWateringDate(lastWateredAt, wateringCycle)

### **테스트 케이스**

| 케이스 | 입력 | 기대값 |
| ----- | ----- | ----- |
| 7일 뒤 계산 | `2026-07-01`, `7` | `2026-07-08` |
| 14일 뒤 계산 | `2026-06-24`, `14` | `2026-07-08` |
| 월 넘어가는 계산 | `2026-07-28`, `7` | `2026-08-04` |
| 연도 넘어가는 계산 | `2026-12-30`, `7` | `2027-01-06` |

### **Given-When-Then**

* Given 마지막 물 준 날짜와 물주기 주기가 주어졌을 때  
* When 다음 물주기 날짜를 계산하면  
* Then `YYYY-MM-DD` 형식의 정확한 날짜가 반환되어야 한다.

---

## **6.2 물주기 상태 계산 테스트**

### **대상 함수**

getWateringStatus(nextWateringAt)

### **기준 날짜**

테스트에서는 오늘 날짜를 `2026-07-07`로 고정한다.

### **테스트 케이스**

| 케이스 | 입력 | 기대값 |
| ----- | ----- | ----- |
| 지연 상태 | `2026-07-06` | `overdue` |
| 오늘 상태 | `2026-07-07` | `today` |
| 예정 상태 | `2026-07-08` | `upcoming` |

### **Given-When-Then**

* Given 오늘 날짜가 고정되어 있을 때  
* When 다음 물주기 날짜를 비교하면  
* Then overdue, today, upcoming 중 하나가 반환되어야 한다.

---

## **6.3 식물 등록 데이터 생성 테스트**

### **대상 함수**

createUserPlant({ preset, nickname, lastWateredAt })

### **테스트 케이스**

| 케이스 | 기대값 |
| ----- | ----- |
| 프리셋 정보 복사 | 식물명, 이미지, 주기, 양, 방식이 복사된다 |
| 사용자 입력 반영 | nickname, lastWateredAt이 반영된다 |
| 다음 날짜 계산 | nextWateringAt이 자동 계산된다 |
| 기록 초기화 | wateringLogs는 빈 배열이다 |
| 생성/수정일 생성 | createdAt, updatedAt이 존재한다 |

---

## **6.4 물주기 완료 처리 테스트**

### **대상 함수**

completeWatering(plant)

### **기준 날짜**

오늘 날짜는 `2026-07-07`로 고정한다.

### **테스트 케이스**

| 케이스 | 기대값 |
| ----- | ----- |
| 마지막 물 준 날짜 갱신 | lastWateredAt \= `2026-07-07` |
| 다음 물주기 갱신 | 주기 7일이면 `2026-07-14` |
| 기록 추가 | wateringLogs에 새 기록 1건 추가 |
| 최신 기록 우선 | 새 기록이 배열 맨 앞에 추가 |
| 수정일 갱신 | updatedAt이 변경됨 |

---

## **6.5 입력값 검증 테스트**

### **대상 함수**

validateAddPlantForm({ selectedPreset, nickname, lastWateredAt })

# **7\. LocalStorage 테스트 설계**

## **대상 함수**

loadStorage()  
saveStorage(storage)  
clearStorage()

## **테스트 케이스**

| 케이스 | 기대값 |
| ----- | ----- |
| 저장값 없음 | DEFAULT\_STORAGE 반환 |
| 정상 JSON 있음 | 파싱된 데이터 반환 |
| 깨진 JSON 있음 | DEFAULT\_STORAGE 반환 |
| version 불일치 | DEFAULT\_STORAGE 반환 |
| 저장 실행 | 지정 key에 JSON 문자열 저장 |

# **8\. UI 테스트 설계**

## **8.1 홈 화면**

### **테스트 케이스**

* 등록된 식물이 없으면 빈 상태 문구가 보인다.  
* 등록된 식물이 있으면 식물 카드가 보인다.  
* 오늘 물 줄 식물은 “오늘 물주기”로 표시된다.  
* 지연 식물은 “물주기 지연”으로 표시된다.  
* 물주기 완료 버튼을 누르면 상태가 갱신된다.

---

## **8.2 식물 등록 화면**

### **테스트 케이스**

* Top5 식물 카드가 표시된다.  
* 식물을 선택하면 기본 물주기 정보가 표시된다.  
* 필수값 없이 저장하면 에러 메시지가 표시된다.  
* 정상 입력 후 저장하면 홈 화면으로 이동한다.  
* 저장된 식물은 LocalStorage에 반영된다.

# **9\. 테스트 우선순위**

## **1순위**

* 날짜 계산  
* 상태 계산  
* 식물 등록 데이터 생성  
* 물주기 완료 처리  
* LocalStorage 복구

## **2순위**

* 입력값 검증  
* 홈 화면 빈 상태  
* 식물 등록 화면 저장 흐름

## **3순위**

* 이미지 fallback  
* 상세 화면 기록 표시  
* 삭제 버튼 동작

---

# **10\. 완료 기준**

테스트 설계 기준에서 MVP는 아래 조건을 만족해야 한다.

* 핵심 유틸 함수 테스트가 모두 통과한다.  
* LocalStorage 예외 상황 테스트가 통과한다.  
* 식물 등록 데이터 생성 테스트가 통과한다.  
* 물주기 완료 처리 테스트가 통과한다.  
* 홈 화면과 등록 화면의 주요 사용자 흐름 테스트가 통과한다.  
* 테스트 대상은 PRD의 MVP 범위를 벗어나지 않는다.
